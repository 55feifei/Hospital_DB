# 医院门诊预约管理系统 — 数据库设计文档

## 1. ER 模型

### 1.1 实体识别

| 实体名     | 类型 | 关键属性                                                       |
|------------|------|----------------------------------------------------------------|
| 科室       | 强   | dept_id (PK), dept_name (UQ), description, location           |
| 医生       | 强   | doctor_id (PK), name, gender, title, fee, intro, status       |
| 患者       | 强   | patient_id (PK), name, gender, birth_date, id_card (UQ), phone (UQ) |
| 账号       | 强   | user_id (PK), username (UQ), password_hash, role, ref_id      |
| 排班       | 弱   | schedule_id (PK), work_date, time_slot, total_quota, remaining_quota |
| 预约       | 关联 | appt_id (PK), appt_no, status, create_time                    |
| 病历       | 弱   | record_id (PK), chief_complaint, diagnosis, prescription, advice |
| 缴费       | 弱   | payment_id (PK), amount, pay_method, pay_time, status         |

### 1.3 联系基数

| 联系               | 基数  | 说明                                            |
|--------------------|-------|-------------------------------------------------|
| 科室 — 医生        | 1:N   | 一个科室多名医生，一名医生属于一个科室           |
| 医生 — 排班        | 1:N   | 一名医生多次排班                                 |
| 患者 — 预约        | 1:N   | 一名患者多次预约                                 |
| 排班 — 预约        | 1:N   | 一次排班接受多个预约（受 total_quota 约束）      |
| 预约 — 病历        | 1:1   | 已就诊预约对应一份病历                           |
| 预约 — 缴费        | 1:N   | 一次预约对应至少一条缴费记录（含可能的退款）     |
| 账号 — 医生/患者   | 1:1   | 登录账号软关联到具体身份                         |

## 2. 关系模式转换

按"实体 → 表，1:N 联系外键放多端，1:1 联系外键放任意一端并加 UNIQUE"规则：

```
DEPARTMENT(dept_id, dept_name, description, location)
DOCTOR(doctor_id, dept_id, name, gender, title, fee, intro, status)
        FK: dept_id → DEPARTMENT(dept_id)
PATIENT(patient_id, name, gender, birth_date, id_card, phone, address)
USER_ACCOUNT(user_id, username, password_hash, role, ref_id, is_active)
SCHEDULE(schedule_id, doctor_id, work_date, time_slot, total_quota, remaining_quota, status)
        FK: doctor_id → DOCTOR(doctor_id)
        UNIQUE(doctor_id, work_date, time_slot)
APPOINTMENT(appt_id, patient_id, schedule_id, appt_no, status, create_time, cancel_time, visit_time)
        FK: patient_id → PATIENT(patient_id)
        FK: schedule_id → SCHEDULE(schedule_id)
        UNIQUE(schedule_id, appt_no)
MEDICAL_RECORD(record_id, appt_id, chief_complaint, diagnosis, prescription, advice, visit_time)
        FK: appt_id → APPOINTMENT(appt_id), UNIQUE
PAYMENT(payment_id, appt_id, amount, pay_method, status, pay_time)
        FK: appt_id → APPOINTMENT(appt_id)
```

## 3. 规范化推导（重点）

### 3.1 初始关系（违反 1NF / 2NF / 3NF 的"挂号大宽表"）

```
APPT_BIG(
    appt_id, 患者姓名, 患者身份证, 患者电话, 患者多个地址,
    医生姓名, 科室名, 科室位置, 医生职称, 挂号费,
    排班日期, 时段, 总号源, 剩余号源,
    诊断列表, 处方列表, 缴费金额, 缴费方式
)
```

### 3.2 第一范式（1NF） — 消除多值与重复组

问题：
- "患者多个地址"、"诊断列表"、"处方列表" 不是原子值

处理：
- 仅保留患者主地址（业务上一次预约只用一个）
- 一次预约的诊断/处方写入同一字段（VARCHAR），多次诊断对应不同 appt_id

得到 1NF：所有属性为原子值。

### 3.3 第二范式（2NF） — 消除非主属性对候选键的部分依赖

候选键暂为 `appt_id`。注意以下函数依赖：
- `患者身份证 → 患者姓名, 患者电话, 患者地址`
- `医生 → 医生职称, 挂号费`
- `科室 → 科室位置`

它们都不依赖于 `appt_id` 本身（appt_id 只是预约这件事的代理键，患者属性独立存在）。
因此把"非完全依赖于主键"的属性分离：

```
PATIENT(patient_id, name, id_card, phone, address)
DOCTOR (doctor_id,  name, title, fee, intro, dept_name)
SCHEDULE(schedule_id, doctor_id, work_date, time_slot, total_quota, remaining_quota)
APPT(appt_id, patient_id, schedule_id, appt_no, status, create_time, diagnosis, prescription, fee_amount)
```

此时所有非主属性完全依赖各表主键，达到 2NF。

### 3.4 第三范式（3NF） — 消除非主属性对主键的传递依赖

剩余问题：
- 在 DOCTOR 中：`doctor_id → dept_name → 科室位置` 存在传递依赖
- 在 APPT 中：`appt_id → diagnosis, prescription` 与 `appt_id → fee_amount` 在业务上语义不同（病历 vs 缴费），且：
  - 一次预约可能多次缴费（部分退款），违反 3NF（fee_amount 多值）
  - 病历独立成对象（如未就诊则无病历）

继续分解：

```
DEPARTMENT(dept_id, dept_name, description, location)
DOCTOR(doctor_id, dept_id, name, title, fee, intro, status)
       FK: dept_id

APPT(appt_id, patient_id, schedule_id, appt_no, status, create_time)
MEDICAL_RECORD(record_id, appt_id, chief_complaint, diagnosis, prescription, advice, visit_time)
PAYMENT(payment_id, appt_id, amount, pay_method, status, pay_time)
```

最终：
- 每张表的主键决定全部非主属性
- 非主属性之间不存在传递依赖
- 已达到 **3NF**

### 3.5 规范化结论

最终模式是 8 张表，主键、外键明晰，无冗余字段、无传递依赖、无插入/删除/更新异常。

| 范式 | 是否满足 | 备注                                       |
|------|----------|--------------------------------------------|
| 1NF  | ✅       | 所有属性原子                                |
| 2NF  | ✅       | 非主属性完全依赖主键（拆分了患者/医生/科室）|
| 3NF  | ✅       | 消除了 doctor → dept_name → location 等传递|
| BCNF | ✅       | 所有决定因素均含候选键                      |

## 4. 表结构详细设计

详见 [`sql/01_schema.sql`](../sql/01_schema.sql)。摘要如下（数据类型选择理由）：

| 字段类型选择       | 适用列                                  | 理由                                              |
|--------------------|-----------------------------------------|---------------------------------------------------|
| INT AUTO_INCREMENT | 各表主键                                | 简洁、有序、空间占用小                             |
| VARCHAR(N)         | 姓名、地址、备注                        | 可变长字符串                                       |
| CHAR(18) / CHAR(64)| 身份证 / 密码哈希                       | 长度固定，便于校验                                 |
| ENUM               | 性别、职称、状态                        | 取值受限，节省空间且自带约束                       |
| DATE / DATETIME    | 出生 / 创建时间                         | 日期 vs 时间戳分离                                 |
| DECIMAL(10,2)      | 金额                                    | 避免浮点误差                                       |

### 9.3 索引设计权衡说明

- 选择 (patient_id, create_time) 作为复合索引而非单列：满足"先按患者过滤再按时间排序"的高频路径，避免 `Using filesort`
- 不在 status 上单独建索引：基数过低（4 个枚举值），区分度差，索引收益小于维护成本
- 不在身份证/手机号建二级索引：UNIQUE 约束已隐式建立索引

## 10. 安全设计

- 应用账号 `hospital_app`：仅 SELECT/INSERT/UPDATE/DELETE/EXECUTE
- 报表账号 `hospital_readonly`：仅 SELECT
- 密码字段使用 SHA-256 哈希存储，永不存明文
- 外键 `ON DELETE RESTRICT`：禁止级联删除关键业务数据
- 触发器 `trg_doctor_before_delete`：业务级删除阻断
- 应用层输入校验（手机号、身份证、用户名）+ DB 层 CHECK 双重保障
