## 1. 引言

### 1.1 编写目的
本文档明确"医院门诊预约管理系统"的功能需求、性能需求、用户角色与业务流程，作为系统设计与实现的依据。读者对象为系统设计人员、开发人员与课程评阅老师。

### 1.2 项目背景
传统的现场挂号方式存在排队长、号源浪费、患者就诊体验差、医院管理粗放等问题。需要建立信息化的门诊预约系统，覆盖患者预约、医生出诊、收费、病历归档、统计分析等核心场景。

### 1.3 术语表

| 术语     | 含义                                                |
|----------|-----------------------------------------------------|
| 排班     | 医生在某一日某一时段（上午/下午）出诊的安排          |
| 号源     | 一次排班可接受的最大预约数（total_quota）           |
| 预约单   | 患者占据某次排班一个号源的记录                      |
| 病历     | 每次就诊产生的诊断、处方、医嘱等记录                |
| 爽约     | 已预约但当日未到诊                                   |

## 2. 用户角色

| 角色   | 职责                                       |
|--------|--------------------------------------------|
| 管理员 | 维护科室、医生、患者、排班；查看统计；运维  |
| 医生   | 查看当日预约、完成就诊、撰写病历            |
| 患者   | 浏览科室/医生、预约挂号、缴费、查看病历     |

## 3. 业务流程

### 3.1 预约挂号流程（主路径）
```
患者登录
   │
   ▼
选择科室 → 选择日期 → 查看可用排班（医生/时段/剩余号源）
   │
   ▼
选定排班 → 系统生成预约单 → 触发器自动减号源 → 触发器生成待支付缴费单
   │
   ▼
患者完成支付 → 等待就诊
```

### 3.2 就诊流程
```
医生登录 → 查看当日预约 → 选定患者 → 填写病历 → 点击"完成就诊"
                                                  │
                                                  ▼
                            UPDATE 状态='已就诊' → 触发器自动建立病历记录
```

### 3.3 取消预约流程
```
患者点击取消 → 调用 sp_cancel_appointment → 行锁校验状态
   │
   ▼
状态机检查：仅"已预约"可取消 → 更新为"已取消"
   │
   ▼
AFTER UPDATE 触发器：恢复排班号源；缴费单标记"已退款"
```

### 3.4 爽约清理流程
```
管理员每日点击"标记爽约"
   │
   ▼
调用游标过程 sp_mark_no_show(date)
   │
   ▼
游标遍历"截止日期前仍处于已预约"的记录 → 逐条 UPDATE 为"爽约" + 备注
   │
   ▼
返回受影响行数
```

## 4. 功能需求

### 4.1 公共功能（FR-0）
- FR-0.1 用户登录鉴权（用户名 + 密码 SHA-256）
- FR-0.2 角色路由（admin / doctor / patient 加载不同面板）
- FR-0.3 输入校验（手机号、身份证、用户名格式）

### 4.2 患者功能（FR-P）
- FR-P.1 浏览科室列表与医生介绍
- FR-P.2 按科室+日期筛选可用排班
- FR-P.3 提交预约（防超约、防重复预约）
- FR-P.4 查看个人预约历史（已预约/已就诊/已取消/爽约）
- FR-P.5 取消未开始的预约
- FR-P.6 查看个人就诊病历

### 4.3 医生功能（FR-D）
- FR-D.1 查看本人当日全部预约（按序号排序）
- FR-D.2 查看患者基本信息
- FR-D.3 完成就诊（标记状态 + 撰写病历）
- FR-D.4 编辑历史病历

### 4.4 管理员功能（FR-A）
- FR-A.1 科室 / 医生 / 患者 / 排班 CRUD
- FR-A.2 查看全部预约清单
- FR-A.3 调用日报统计存储过程，按科室聚合
- FR-A.4 一键标记爽约（游标过程）
- FR-A.5 查看医生工作量视图、最忙医生

## 5. 非功能需求

| 类别       | 要求                                                                |
|------------|---------------------------------------------------------------------|
| 性能       | 单次预约响应 < 500 ms；患者历史查询 < 300 ms                        |
| 并发       | 同一排班 N 个客户端同时抢最后 1 号 → 仅 1 个成功，其余收到"号源已满" |
| 安全       | 密码哈希存储；应用层使用最小权限账号 hospital_app；外键防误删       |
| 可维护性   | 三层分层架构；DDL/DML 全部脚本化，可重建                            |
| 可移植性   | 仅依赖 MySQL 8.0+ 标准特性；Python 3.9+；Windows / Linux 双平台     |

## 6. 数据需求

### 6.1 实体识别（详见数据库设计文档 §1）
- **科室、医生、患者**：基础主体
- **排班**：医生与日期的弱实体
- **预约**：患者与排班的关联实体
- **病历**：预约的扩展实体（一对一）
- **缴费**：预约的扩展实体（一对多，支持退款）
- **账号**：登录主体，软关联医生/患者

### 6.2 数据约束（详见 sql/01_schema.sql）
- 患者身份证 18 位 UNIQUE
- 患者手机号 UNIQUE
- 排班 (doctor_id, work_date, time_slot) 复合唯一
- remaining_quota ≥ 0 且 ≤ total_quota
- 预约 (schedule_id, appt_no) 唯一
- 一次预约最多对应一份病历

## 7. 接口需求

- **数据库接口**：MySQL 协议（PyMySQL，3306 端口）
- **存储过程接口**：`sp_book_appointment` / `sp_cancel_appointment` / `sp_daily_statistics` / `sp_mark_no_show` / `fn_get_doctor_workload`
- **GUI 接口**：PyQt5 桌面客户端，事件驱动
