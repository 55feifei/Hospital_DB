## 四、索引性能基准（实测数据）

测试方法：在 11,523 行 appointment / 6,060 行 schedule / 11,523 行 payment 数据集上，分别进行：
1. 用 `EXPLAIN` 看索引存在时的执行计划
2. `DROP INDEX` 后再次 `EXPLAIN` 并测时（50 次平均）
3. 重建索引

### 4.1 idx_appt_patient_date — 患者历史查询

```sql
SELECT * FROM appointment
WHERE patient_id=1 AND create_time > '2026-01-01'
ORDER BY create_time DESC LIMIT 50;
```

| 状态     | type   | key                     | rows scanned | Extra                              | 平均耗时 |
|----------|--------|-------------------------|--------------|------------------------------------|----------|
| 无该索引 | ref    | (外键 idx_fk_patient_tmp) | 384          | Using where; **Using filesort**    | 14.49 ms |
| 有该索引 | range  | **idx_appt_patient_date** | 384         | Using index condition; **Backward index scan** | 11.10 ms |

**结论**：复合索引消除了 `filesort`，直接以反向索引扫描完成排序，时间下降 ~23%。

### 4.2 idx_payment_status_time — 待支付订单列表

```sql
SELECT * FROM payment
WHERE status='待支付'
ORDER BY created_at DESC LIMIT 50;
```

| 状态     | type | key                       | rows scanned | 平均耗时 |
|----------|------|---------------------------|--------------|----------|
| 无该索引 | ALL  | NULL                       | **10,810**   | 19.92 ms |
| 有该索引 | ref  | **idx_payment_status_time** | **5,405**   | 14.85 ms |

**结论**：从全表扫描升级到 ref 索引访问，扫描行数减半，时间下降 ~25%。

### 4.3 idx_schedule_date_remain — 可用排班查询

```sql
SELECT * FROM schedule
WHERE work_date BETWEEN CURDATE() AND DATE_ADD(CURDATE(), INTERVAL 30 DAY)
  AND remaining_quota > 0;
```

| 状态     | type  | key                        | rows scanned | 平均耗时 |
|----------|-------|----------------------------|--------------|----------|
| 无该索引 | ALL   | NULL                        | **6,060**    | 27.68 ms |
| 有该索引 | range | **idx_schedule_date_remain** | **930**     | 26.62 ms |

**结论**：扫描行数从全表 6060 降至范围 930（**1/6.5**），EXPLAIN 类型从 ALL 升级到 range。

### 4.4 索引选择权衡说明

- 优先选择**非外键列起始**的复合索引，避免与外键自动索引冗余，且能自由 DROP/CREATE 用于性能验证。
- 不在 status 单列建索引：基数过低，区分度差。
- 不在 username/id_card/phone 上建二级索引：UNIQUE 约束已隐式建立。

## 六、测试中发现并修复的 Bug

| #  | 严重性 | 现象                                                   | 根因                                                        | 修复                                                       |
|----|--------|--------------------------------------------------------|-------------------------------------------------------------|------------------------------------------------------------|
| 1  | 阻塞   | 登录报 RuntimeError: 'cryptography' package is required | MySQL 8 默认 `caching_sha2_password` 认证需要 cryptography  | 在 requirements.txt 中加入 cryptography                    |
| 2  | 严重   | `cancel()` 返回值 `msg` 是整数而非字符串                | `Database.call_proc` 返回的 `out` 数组按"IN+OUT"原始顺序，service 取索引错（取了 IN 参数） | 修正 service 层取真实 OUT 索引：book→[2,3]，cancel→[1]，mark_no_show→[1] |
| 3  | 中     | `test_unique_id_card` 失败                              | 测试硬编码的身份证不在种子数据中                            | 改为先 `SELECT` 一个已存在的 id_card，再尝试重复插入        |
| 4  | 中     | `ALTER TABLE … DROP INDEX …` 报 "needed in a foreign key constraint" | 二级索引以外键列开头，被外键依赖                          | ① 重设计索引：`idx_schedule_date_remain` / `idx_payment_status_time` 改用非外键列起始；② 性能测试中先建临时外键索引再删主索引 |

## 七、备份还原演练

```bash
# 1. 备份
mysqldump -u root -p --routines --triggers --single-transaction \
          --default-character-set=utf8mb4 \
          hospital_db > hospital_backup.sql

# 2. 删库重建
mysql -u root -p -e "DROP DATABASE hospital_db; CREATE DATABASE hospital_db CHARACTER SET utf8mb4;"

# 3. 还原
mysql -u root -p hospital_db < hospital_backup.sql

# 4. 验证
mysql -u root -p -e "USE hospital_db; SELECT COUNT(*) FROM appointment;"
```

实测：备份耗时 ~1.2 s，还原 ~3.5 s，appointment 行数前后一致。
